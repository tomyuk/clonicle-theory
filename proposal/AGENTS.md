# AGENTS.md

## 適用範囲

この文書は、`proposal/` 配下で作成・編集する提案書、ホワイトペーパー、事業構想書、役員説明資料、および関連Markdown文書に適用します。下位ディレクトリに個別の `AGENTS.md` がある場合は、より具体的な指示を優先します。

## 基本文体

- 本文は原則として日本語の「です・ます調」で統一します。
- 論文調の「である調」は、引用、定義文、憲章、宣言、数式説明など、意図的に硬質な表現が必要な場合に限ります。
- 読者は経営者、役員、士業、金融機関、自治体、教育機関を想定します。理論的な深さを保ちながら、専門外の読者にも読み進められる文章にします。
- 読みやすさを理由に文章を細切れにしません。チャット的な短文の連続を避け、一段落の中で一つの論旨を展開します。

## 改行と段落

- 見出し以外では、文ごとの改行を行いません。
- 空行は、論点、視点、時間軸、対象、または議論の段階が切り替わる箇所に限ります。
- 一段落は目安として200〜700字程度とし、短い一文だけの段落を連続させません。
- 強調のための一文独立は、章の中心命題、問い、定義、結論などに限定します。
- 箇条書きは、役割、要件、比較軸、手順など、列挙そのものに意味がある場合だけ使用します。通常の論証を箇条書きへ逃がしません。

## 論理構成

- 基本構造は「問題提起 → 定義 → 論証 → 含意 → 次の論点」です。
- 一章につき、中心となる主張は原則一つに絞ります。
- 主張を繰り返すだけでなく、その主張が必要となる理由、成立条件、反例や限界を示します。
- 「次章では〜」という説明的な予告を多用せず、現在の章の帰結から次の問いが自然に立ち上がる構成を優先します。
- 新しい概念を導入する際は、既存概念との差異、適用範囲、誤解されやすい点を明示します。

## Chronicle First

- 企業、組織、人材、AI、士業、金融機関、地域社会を論じる際は、クロニクルを単なる履歴や年表として扱いません。
- クロニクルは、主体が時間の中で積み重ねてきた経験、判断、関係性、学習、価値が、一つの意味体系として連続し、更新される動的構造として扱います。
- 関係性、学習、価値、文脈をクロニクルから安易に分離し、後付けの下位概念として扱いません。
- アイデンティティは、クロニクルから切り離された独立実体ではなく、クロニクルが内外に現れた姿として扱います。
- AI導入、業務改革、人材育成、資本評価などの議論は、まず企業のクロニクルを理解し、何を保存・更新・継承するかを問うところから始めます。

## Brainful関連概念

- Brainfulは単なる知能水準、能力分類、AI製品、研修手法として記述しません。
- Brainfulは、クロニクルを理解し、その価値を未来へ発展させる継続的な営みとして扱います。
- Brainful Protocolは、異なる主体を中央集権的に統合する制度ではなく、独立性と専門性を保ったまま協働するための共通原理・共通作法として扱います。
- Brainful Allianceは固定的な組織や販売代理網ではなく、Brainful Protocolを共有した主体間に形成される協働状態・実践共同体として扱います。
- 士業や金融機関は単なる販路や紹介者ではなく、企業のクロニクルを異なる角度から観測し、支え、共創する主体として位置付けます。

## RDE差異検証

生成・改稿時には、元の議論との意味差分を次の観点で確認します。

1. 保存された要素
2. 変換された要素
3. 補完された要素
4. 未解決の要素
5. 逸脱リスク
6. 次回更新方針

特に、次の逸脱を避けます。

- 元の主張より強い断定へ変えていないか。
- 理論的提案や仮説を、実証済みの事実として記述していないか。
- 思想的射程を、実装上の都合だけで狭めていないか。
- 実装上の便宜を、理論上の必然へすり替えていないか。
- クロニクルに本来含まれる関係性・時間性・学習を、別概念として外部化していないか。

## 用語と表記

- 英語概念は初出時に日本語の簡潔な説明を添えます。
- `Chronicle`、`Brainful`、`Brainful Protocol`、`Brainful Alliance`、`Chronicle Capital` など中核概念の表記は文書内で統一します。
- 日本語の「クロニクル」と英語の `Chronicle` は、文脈に応じて使い分けますが、同一段落内で無意味に混在させません。
- 略語は初出時に正式名称を示します。
- 強調は必要最小限にし、太字を連続させません。

## 図表

- 図は本文の代替ではなく、本文の論理を圧縮・補強するものとして使用します。
- Mermaidを用いる場合、ノード内の複数語ラベルはダブルクォートで囲みます。
- Mermaidの `[]` 内テキストはダブルクォートで囲みます。
- 3段を超える関係は、原則として `flowchart TD` を用いて縦方向に整理します。
- 図の前後には、その図が何を示し、どの主張を支えるかを文章で説明します。
- 図表には番号と短いタイトルを付け、本文から参照できるようにします。

## 事実性と引用

- 外部の事実、統計、法制度、制度名称、時点依存の情報は、公開前にファクトチェックします。
- 理論的提案、独自概念、仮説は、その性格を明示します。
- 引用は出典を明示し、原文の意味を歪めません。
- 参考文献は本文の装飾ではなく、主張の根拠または理論的背景として選定します。

## Markdown

- 見出し階層は `#` から順に飛ばさず使用します。
- 本文内でのHTMLは、Markdownだけでは表現困難な場合に限定します。
- 一行の過度な長文化は避けますが、文単位の強制改行は行いません。
- コード、設定、数式、Mermaidは適切なフェンス付きコードブロックを使用します。
- 文書冒頭にメタデータを置く場合はYAMLフロントマターを使用し、`author` は原則 `Tomoyuki Kano` とします。

## 品質基準

完成稿は、次の状態を目指します。

- 読みやすいが、単純化しすぎていない。
- 理論的だが、研究論文のように閉じていない。
- 経営実務へ接続できるが、単なる手順書に落ちていない。
- AIを主役にせず、企業のクロニクルと人間の判断を中心に据えている。
- 数年後に読んでも、特定ツールの流行に依存して古びない。
